The risk review found that `_review_count` currently tracks only newly classified low-confidence comments, not cached low-confidence comments. That means the label `need review` is misleading today.

Required behavior for this plan:
- `need_review` means **all comments encountered in the classify run that require review**, including cached low-confidence items
- Make the classifier contract compute that intentionally, not by wording around the current undercount

Implementation notes:
1. In `_classify_comment()`, when a cache hit is returned with `cached_confidence < 0.75`, increment the same review counter used for newly classified low-confidence items, or equivalently compute `need_review` from returned `ClassifiedComment.needs_review` values across the run
2. Add `get_summary_counts()` that returns exactly `{"new": ..., "cached": ..., "need_review": ..., "failed": ...}`
3. Update `_run_classify()` to use that helper and return the honest string:
```python
return (
    f"Classified {new} new, {cached} cached, "
    f"{need_review} need review, {failed} failed."
```

**Make D-12 explicit instead of assumed:**

Treat these as exit code `1`:
- Missing config / missing API key
- No classified input is available (`classified-pr-*.json` is absent in the configured cache dir)
- Upstream generator surfaces a no-input / no-cache failure via explicit exception
- `generate_all()` returns **zero successful outcomes** and at least one failure

Before constructing `KBGenerator()`, add a CLI-owned preflight check for classified input so D-12 does not depend on generator internals that 07-02 did not promise:
```python
from github_pr_kb.generator import DEFAULT_CACHE_DIR, KBGenerator

if not list(DEFAULT_CACHE_DIR.glob("classified-pr-*.json")):
    raise click.ClickException(
        "Generation failed: no classified PR cache found.\n"
        "Hint: run `github-pr-kb classify` before `github-pr-kb generate`."
    )
```

This preflight is part of 07-03 because the requirement is CLI exit-code behavior, and it avoids treating an empty/no-input run as a successful no-op.

For this plan, define:
```python
successful_outcomes = result.written + result.skipped + result.filtered
```

Then:
```python
if result.failed and successful_outcomes == 0:
    raise click.ClickException(
        f"Generation failed: {len(result.failed)} article(s) failed and no output was produced."
    )
```

Treat these as exit code `0`:
- Any run with at least one successful outcome, even if some articles failed

In that partial-failure case, emit a warning to stderr:
