# Phase 9: Wiki-style KB Synthesis - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the `generate` command with a two-pass topic synthesis post-step. After classification, the generator groups related per-comment articles into topic pages organized by category. Topic pages replace per-comment articles as the primary KB output. Per-comment articles are intermediate artifacts that exist only in memory. The pipeline remains `extract → classify → generate` with no new CLI commands.

</domain>

<decisions>
## Implementation Decisions

### Topic Page Structure
- **D-01:** Sources are cited inline within the topic body - PR references woven naturally into text (e.g., "This was adopted in PR #12 and later revised in PR #45"). No separate Sources footer section.
- **D-02:** Topic pages reuse the same category-specific templates from Phase 7 (gotcha → Symptom/Root Cause/Fix, architecture_decision → Context/Decision/Consequences, etc.). Consistent format across per-comment and topic articles.
- **D-03:** Topic frontmatter is reader-focused and minimal: `title`, `category`, `last_updated`, `needs_review`. No source counts, comment IDs, or PR links in frontmatter - all provenance data lives in the manifest only.
- **D-04:** Single-source topics are synthesized identically to multi-source topics - same template, same format. No special handling or pass-through for single-source topics.

### Synthesis Behavior
- **D-05:** Synthesis runs automatically every time `generate` is called. A `--no-synthesize` flag provides an escape hatch for per-comment-only output (debugging, cost-sensitive runs).
- **D-06:** Per-comment articles are NOT written to disk. They exist only in memory as intermediate input to the topic planning pass. The final KB directory contains only topic pages.
- **D-07:** Both the topic planning pass and synthesis pass use the same model - the existing `ANTHROPIC_GENERATE_MODEL` setting. No new environment variable. Users can already override the model via this setting.

### Manifest & Migration
- **D-08:** The `.manifest.json` is extended with nested sub-keys: `{"comments": {id: topic_slug}, "topics": {slug: {sources, content_hash, last_synthesized}}}`. One file, two sections.
- **D-09:** Old flat-format manifests (no `"comments"` key) are auto-detected and migrated on first run: existing dict is wrapped under `"comments"`, empty `"topics"` key is added, then a full topic synthesis runs for all articles. Old per-comment articles are replaced by topic pages.

### Cross-references & Index
- **D-10:** Cross-references between topic pages are generated by Claude during synthesis. The synthesis prompt receives the full list of topic slugs from the planning pass so Claude knows what's linkable. Standard markdown links `[text](../category/slug.md)`.
- **D-11:** A post-synthesis validation pass strips broken cross-reference links (converts to plain text, keeping display text). Prevents 404s in the KB.
- **D-12:** INDEX.md keeps the same format (## Category (count), bulleted links with summaries) but links to topic pages instead of per-comment articles. The `title` from topic frontmatter is used as the display text.

### Claude's Discretion
- Exact synthesis prompt wording and max_tokens for topic pages
- Topic planning prompt wording and JSON structure constraints
- How to handle topics that span multiple categories (assign to most common)
- Content hash implementation details (SHA-256 of sorted source bodies)
- How chronological evolution is phrased in topic bodies

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `KBGenerator._build_article()` - Current per-comment synthesis with Claude. The topic synthesis will follow a similar pattern (prompt → API call → extract body → write).
- `_CATEGORY_SECTIONS` dict - Category-specific template sections. Reusable for topic pages (D-02).
- `SYNTHESIS_SYSTEM_PROMPT` - Existing synthesis system prompt. Topic synthesis will need its own variant that handles multi-source merging and chronological awareness.
- `_write_atomic()` - Atomic file write pattern. Reuse for topic page writes.
- `slugify()` - URL-safe slug generation. Reuse for topic page filenames.
- `_extract_synthesized_body()` - Parse Anthropic response. Reuse for topic synthesis responses.
- `_looks_like_source_echo()` - Source echo detection. May need adaptation for multi-source topics.
- `GenerateResult` model - Can be extended with synthesis-specific counts (topics_written, topics_skipped).

### Established Patterns
- Lazy settings import inside `__init__` to prevent import-time ValidationError
- `ConfigDict(extra='ignore')` on all Pydantic models
- Manifest-based dedup with JSON round-trip (str keys for comment_id)
- Transactional generation with staging dir + rename (`_generate_all_transactionally`)

### Integration Points
- `_run_generation_pass()` - Synthesis post-pass hooks in after this method. Per-comment articles become in-memory intermediates feeding the topic planner.
- `_load_manifest()` / `_save_manifest()` - Must handle new nested format and old flat format migration.
- `_generate_index()` - Must read topic page frontmatter (title field) instead of per-comment article headings.
- `_parse_article_metadata()` - Currently parses hand-rolled YAML frontmatter. Research recommends switching to `python-frontmatter` library for robustness.

</code_context>

<specifics>
## Specific Ideas

- Frontmatter is for the reader, not for bookkeeping - no internal IDs or source counts in topic page frontmatter
- Per-comment articles never touch disk - the KB directory only contains topic pages
- Phase 7's category templates are proven and should be reused, not redesigned
- Chronological evolution handled naturally in synthesis prompt, not as a separate template section

</specifics>
